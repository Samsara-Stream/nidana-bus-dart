- The review document said "The bus creates subjects, activates subscriptions, disposes them, generates message IDs, stamps wall-clock timestamps, and may intercept traffic. Those are runtime effects, not pure computation.". The "pure" part is from the library users' perspective. The library itself is enabling the library users' to write "pure" code in the core part. I don't think it's possible to have the bus (library) part to be "pure". Maybe this is something that needs to be clarified.
- The review document says "topologies are pure and contain no side effects, but examples and combinator guidance imply that async work may happen inside topology pipelines." Again this is a misunderstanding on the executor (the library) and the "purity" that is defined by the user of the library. The executor itself runs with side-effects, but the behavior is well-defined. I'm not sure how to frame this, but I guess some clarity can be added.
- On "5. Message envelope propagation is underdefined". I'm not sure if the concerns mentioned in there are valid. For instance, "How is correlationId assigned for root messages?". I think this is trivial. The library simply generates the ID when a new envelope is created. "How is causationId computed when one output depends on multiple inputs, such as combine, zip, or withLatest?" The library can always intercept the operation to compute. I mean, I'm pretty sure there is a way, but I wouldn't call it "hard part". But CMIIW.
- On "7: No privileged observers conflicts with bus-level interception". Again this is the same misunderstanding on the meaning of privileged observer. Of course the library is privileged. From the library user's perspective, when they create something, be it service or UI, nothing is privileged.
